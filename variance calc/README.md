# Telegram MTT Variance Bot

Telegram-бот на Python для расчета variance в MTT через аналитическую модель и Monte Carlo симуляцию. Бот работает прямо в Telegram-чате и возвращает краткое резюме, графики, статистику, оценку банкролла и превью структуры выплат.

## Возможности

- Команды `/start`, `/help`, `/variance`, `/example`
- Пошаговый FSM-сценарий сбора параметров турнира
- Автоматическая генерация ступенчатой структуры выплат
- Минимальный кэш строго `1.5` net buy-in
- Monte Carlo симуляция с фиксированным внутренним размером выборки `5000`
- График случайных траекторий
- Гистограмма итоговых результатов с нормальным приближением
- График структуры выплат
- Оценка EV, SD, доверительных интервалов и вероятности минуса
- Рекомендации по банкроллу и оценка risk of ruin

## Как работает модель выплат

Бот не копирует точную payout-структуру конкретного покер-рума. Вместо этого используется синтетическая аппроксимация, вдохновленная стандартными MTT-лестницами и референсными скриншотами.

Экономическая модель:

- `B` = полный бай-ин игрока с учетом рейка
- `r` = `rake_percent / 100`
- `net_buyin = B / (1 + r)`
- `prize_pool = N * net_buyin`
- `M = max(1, round(N * paid_percent / 100))`

Минимальный кэш:

- `min_cash = 1.5 * net_buyin`

Сначала резервируется:

- `base_pool = M * min_cash`

Если `base_pool > prize_pool`, бот возвращает дружелюбную ошибку о несовместимости выбранного `% ITM` с минимальным кэшем `1.5` бай-ина.

Оставшаяся часть призового фонда распределяется top-heavy:

1. Для каждого оплачиваемого места строится raw weight:

```text
alpha(N) = 1.05 + 0.09 * ln(N)
base_weight(k) = 1 / (k - 1 + 0.7) ^ alpha(N)
```

2. Топ-3 усиливается отдельными множителями:

```text
1 место -> 1 + 0.22 * ln(N)
2 место -> 1 + 0.12 * ln(N)
3 место -> 1 + 0.06 * ln(N)
```

3. После топ-10 кривая переводится в bucket/step ladder:

- `11-15`
- `16-20`
- `21-25`
- `26-30`
- `31-35`
- `36-40`
- `41-50`
- `51-60`
- `61-75`
- `76-100`
- `101-125`
- `126-150`
- далее шаг расширяется логично

4. Внутри каждого bucket все места получают одинаковую выплату. Это делает структуру ступенчатой, а не гладкой.

5. Выплаты округляются до центов, остаток после округления добавляется к последнему оплачиваемому месту, чтобы итоговая сумма выплат совпала с полным prize pool.

## Модель finish probabilities

Распределение финишей внутри ITM строится отдельно:

```text
finish_weight(k) = 1 / (k - 1 + 0.7) ^ 0.65
pi_k = finish_weight(k) / sum(finish_weight)
```

Далее из целевого ROI решается общая ITM-вероятность:

```text
t = B * (1 + ROI) / sum(pi_k * payout_k)
```

Если `t > 1`, бот возвращает понятную ошибку, что ROI слишком высокий для выбранной структуры выплат.

## Monte Carlo движок

Monte Carlo используется как основной визуальный слой.

- Внутренний размер симуляции зафиксирован: `5000`
- Пользователь не вводит sample size
- Для каждого из `5000` прогонов моделируется блок из `T` турниров
- Результат турнира:
  - `-B`, если игрок не попал в деньги
  - `payout_k - B`, если занял оплачиваемое место

Симулятор сохраняет:

- итоговые результаты по всем прогонам
- кумулятивные траектории
- выборку траекторий для графика
- требуемый банкролл по каждому прогону

## Аналитические метрики

Также считаются аналитические значения:

- `EV`
- `SD`
- интервалы `70%`, `95%`, `99.7%`
- вероятность завершить дистанцию в минусе

Используются формулы:

```text
mu = B * ROI
E[X^2] = (1 - t) * B^2 + sum(pk * (payout_k - B)^2)
Var(X) = E[X^2] - mu^2
sigma = sqrt(Var(X))
```

Для дистанции `T`:

```text
mu_T = T * mu
sigma_T = sqrt(T) * sigma
```

## Bankroll и risk of ruin

Оценка банкролла строится симуляционно:

- Для каждого прогона считается максимальная просадка
- Требуемый банкролл = величина, которая покрывает эту просадку
- Для целевых уровней `50%`, `15%`, `5%`, `1%` берутся соответствующие квантили

Если пользователь ввел свой банкролл, бот показывает:

- текущий банкролл
- оценку риска разорения
- долю прогонов без падения ниже нуля
- предупреждение, если банкролл ниже умеренно-консервативного уровня

## Ограничения и допущения

- Структура выплат синтетическая, а не room-specific
- Модель finish probabilities калибруется под заданный ROI
- Симуляция фиксирована на `5000` прогонов
- Для очень агрессивных ROI бот может отказать в расчете, если модель требует невозможную ITM-частоту

## Run two bots with separate tokens

1. Create env files from templates:
   - copy .env.bot1.example .env.bot1
   - copy .env.bot2.example .env.bot2
2. Put different BOT_TOKEN values into each file.
3. Start bot instance 1:
   - powershell -ExecutionPolicy Bypass -File .\start_bot.ps1 -EnvFile .env.bot1
4. Start bot instance 2 in another terminal:
   - powershell -ExecutionPolicy Bypass -File .\start_bot.ps1 -EnvFile .env.bot2

Important: one token must be used by only one polling process.
